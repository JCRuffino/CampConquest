# Camp Conquest — Code Review & Improvement Plan

A prioritised work plan from a full review of the codebase (2026-07-15). Written to be
handed to an AI coding agent (or a human). **Read the constraints first.**

## Context for the implementer

- Vanilla JS ES modules, no build step, no dependencies beyond Leaflet (CDN) and the
  Firebase web SDK (CDN import in `firebase.js`). **Keep it that way** — the app is
  served as static files and must run from GitHub Pages / `python -m http.server`.
- All shared state lives in one Firebase RTDB node (`camp/<code>/gameState`), mutated
  exclusively through `mutateState` transactions in `firebase.js`. Every game action in
  `actions.js` re-validates against a `{owner, locked}` snapshot (`expected`) so
  concurrent actions abort instead of clobbering. Preserve this pattern.
- **Do not change game rules.** The rules are deliberate and were iterated with the
  game designer. If a fix seems to require a rule change, stop and flag it instead.
- Area names are database keys (`toKey(name)`). Renaming an area is a breaking change
  requiring a game reset.
- Changing the shape of `gameState` is acceptable **before the event** but note in the
  PR that an admin "Reset Game" is required after deploying.
- `areas.js` is generated by `tools/generate_areas.py` (see P2-1 — commit it first).
  Never hand-edit polygon coordinates in `areas.js`; change the generator inputs.
- Manual test loop: `py -m http.server`, open an admin tab (unlock in Settings, save
  Game Setup) plus two browser tabs with `?code=<test code>` that each claim a team
  from the picker. Note: tabs in the same browser profile share one anonymous-auth
  uid — use different profiles/incognito for distinct "phones".

## P0 — Bugs (fix first)

### P0-1: Game reset wipes team names and player rosters
`main.js defaultState()` contains only `areas` + `attempts`; the reset callback pushes
it wholesale, so `teamNames`, `players`, and `attemptTurn` vanish. Devices remain
"joined" via localStorage, so nobody is re-prompted — phone-duty messages degrade to
"Player 1/Player 2" and custom team names revert silently. On game day the admin will
set everything up, then reset to start fresh, and lose it all.
**Fix:** preserve `teamNames` and `players` across reset (copy from current state into
the fresh state inside the reset flow); reset `attemptTurn` to zero. Acceptance: set
team + player names, reset game, names survive; areas/attempts/log are cleared.

### P0-2: Blind-steal verdict uses confirm() where Cancel is destructive
`map.js` claim handler (steal branch): after entering a result, `window.confirm` asks
"Did you BEAT it?" — OK steals+locks, **Cancel locks the area for the owner forever**.
Users instinctively press Cancel/back-gesture to mean "abort", and some mobile browsers
dismiss dialogs (= Cancel) on backgrounding. An accidental tap permanently settles an
area.
**Fix:** replace with a non-dismissable in-app modal with two explicit buttons ("✅ We
beat it — steal & lock" / "🛡️ We fell short — locks for <owner>") and no
click-outside-to-close; dismissing via Escape/back should return to the popup with the
result preserved, not resolve the duel. Acceptance: no single accidental gesture can
resolve a steal; abandoning the dialog leaves the attempt in progress.

### P0-3: Player names silently lost if Firebase isn't connected at join time
`settings.js` join handler calls `mutateState` to store names; `mutateState` aborts when
the state node is null/unloaded (returns uncommitted) and nothing retries. The device
still joins the team locally.
**Fix:** if the write doesn't commit, either retry once state arrives (stash pending
names and flush in the gameState listener) or block joining with a "still connecting"
message. Acceptance: join a team immediately after a cold load on a slow connection;
names appear in the next attempt's phone-duty message.

### P0-4: `startAttempt` / `failChallenge` don't check `gs.winner` inside the transaction
`claimArea` re-checks `gs.winner` inside its mutator; the other two only call
`gameOverGuard(gameState.data)` against the possibly stale local cache. A client that
hasn't yet received the winner update can start/fail attempts after the game has ended.
**Fix:** add the same `if (gs.winner) { failReason = …; return; }` guard inside both
mutators. Acceptance: with a winner set in the DB but a stale client cache, both
actions abort with a sensible message.

### P0-5: Open popups are stale snapshots
`handleAreaClick` builds popup DOM once. If the remote state changes while it's open
(another team claims/steals/starts, admin edits), the popup keeps showing old status
and offering old actions. The transaction snapshot guards prevent corruption (the
action fails with "this area just changed — reopen it"), but the UX is confusing and
players will hit that error often on a busy board.
**Fix:** in `updateAreaLayers` (or a dedicated listener), detect if the open popup's
area changed (compare a serialised `{owner, locked, attemptingBy, failedBy, result}`
signature captured at popup build time) and re-render the popup in place (call
`handleAreaClick` again at the same latlng) — except while the local team is
mid-resolution (result prompt open). Acceptance: with a zone popup open in tab A,
claim it from tab B; tab A's popup updates within a second without user action.

## P1 — Robustness / UX

### P1-1: Replace all window.prompt/confirm/alert with in-app modals
Used for: game code entry, player names, start/claim/fail confirmations, result entry,
steal verdict, editor prompts, admin reset. Native dialogs are ugly, block JS, can't be
styled, lose input on accidental dismiss, and are suppressed in some Android WebView /
iOS standalone-PWA contexts. `index.html` still ships unused `.modal-overlay`/
`.modal-box` CSS from an earlier iteration — build a small `modal.js` helper
(`showModal({title, body, fields, buttons}) → Promise`) on top of it and migrate all
call sites. Do P0-2 as part of this. Acceptance: no `window.prompt|confirm|alert` left
in the codebase (grep), all flows keyboard/touch friendly.

### P1-2: Stuck attempts need visibility and an expiry story
An unresolved attempt blocks the zone for everyone (`attemptingBy`) indefinitely; only
an admin popup "Apply" clears it, and nothing surfaces which zones are stuck. Add:
(a) attempt age shown in the zone popup ("started 14 min ago"); (b) an admin-only line
in the popup with a one-tap "clear stuck attempt" (uses existing `adminSetArea` with
unchanged fields, or a new lighter action); (c) optional: auto-expiry after N minutes
(config constant, default off — **rule-adjacent, confirm with the owner before
enabling**). Acceptance: admin can see and unstick a stale attempt in two taps.

### P1-3: Timer end should announce final standings
`maybeLogGameEnd` logs a generic "check the leaderboard". Compute the final ranking
(`rankTeams`/`getScores`) in the same place and log e.g. "🏁 GAME OVER — Team X wins
with 9 pts (Y 7, Z 5)". Do **not** set `gs.winner` (that field means instant win and
blocks admin corrections differently). Acceptance: at countdown end every device
toasts the winner and scores.

### P1-4: Field conditions — connectivity and screen sleep
Woodland site, patchy signal, phones in pockets. (a) Add a PWA manifest + icons +
`apple-mobile-web-app-capable` meta so teams can add to home screen and run fullscreen;
(b) request a screen wake lock (`navigator.wakeLock`) while an attempt started by this
device's team is unresolved, releasing after; (c) show data freshness — track the last
`onValue` arrival time and switch the pill to "🟡 Reconnecting…" when >30s stale
(RTDB's in-memory queue means writes made while offline flush on reconnect, but reads
go stale silently). Acceptance: app installable, timer screen stays awake during an
attempt, stale connection visibly flagged.

### P1-5: Mobile input niggles
Inputs with font-size <16px trigger iOS Safari auto-zoom (several are 13px). Bump form
inputs to ≥16px or add `maximum-scale=1` thoughtfully (prefer the font-size fix).
Check the admin/password and game-code inputs specifically. Acceptance: focusing any
input on iOS doesn't zoom the page.

## P2 — Code quality / maintainability

### P2-1: Commit the zone-geometry generator into the repo
`areas.js` is generated by a Python script that currently lives outside the repo (in a
Claude Code scratchpad, `make_zones5.py`). If it's lost, the zone pipeline (linked
zones touch / unlinked gap / no overlap / weighted four-corner splits) can't be re-run.
Add it as `tools/generate_areas.py` with the boundary JSON it reads
(`tools/bushywood-boundary.json`), plus a README section: inputs (traced polygons,
connections list, per-zone expansion factors, weighted `SHIFT` pairs), how to run, and
the rule that `areas.js` is generated output. Acceptance: `py tools/generate_areas.py`
reproduces the committed `areas.js` byte-for-byte (modulo trailing newline).

### P2-2: Extract CSS; remove dead styles and dead code
Move the ~400-line `<style>` block from `index.html` to `styles.css`. Delete unused
rules (`.challenge-card`, `.card-*`, `.held-*`, `.no-challenges`, `.reset-blocked-msg`,
`.modal-*` if P1-1 rebuilds them, `.divider`, `.section-title` — verify each with
grep). Remove dead JS: `isEditorActive` (unexported use), `findArea` in `shared.js` if
unused, the `linkSeams` export in `areas.js` generator output if not consumed. Replace
repeated inline `style="…"` strings in JS-built HTML with classes where practical.
Acceptance: no functional change (verify with screenshots), file sizes down, grep
finds no orphan selectors.

### P2-3: Split `map.js` (~650 lines)
It currently holds: map/tile init, hatch patterns, area layers + styling, attempt
badges, the entire popup state machine, the vertex editor, and player-location
sharing. Split into `map.js` (init/layers/badges), `popup.js` (the
`handleAreaClick` state machine), `editor.js` (vertex/midpoint editor). Keep the
public API used by other modules (`initMap`, `addAreas`, `updateAreaLayers`, `getMap`,
`toggleAreaEditor`) stable via re-exports. Acceptance: identical behaviour; no module
import cycles (actions ↔ popup is the risk — popup imports actions, never the reverse).

### P2-4: Unit tests for the pure game logic
`shared.js` is almost all pure functions: `getScores` (set bonuses, unique-max
most-connected), `largestCluster`, `instantWinner` threshold, `rankTeams` tie-breaks,
`fixArrays`, `findArea`/`toKey`, `formatCountdown`, `normalizeGameCode`. Add
`node --test` tests (no framework install; they must not require a browser — guard the
`localStorage` accessors or inject). Cover the fiddly cases: bonus tie awards nobody;
threshold with 21 areas = 11; cluster counting ignores locked/unlocked distinction;
`fixArrays` converting Firebase objects back to arrays. Acceptance: `node --test tests/`
green; tests run without network.

### P2-5: Server timestamps for the log
`pushLog` uses client `Date.now()`; a phone with a skewed clock makes history ordering
and the toast `timestamp > lastToastTs` filter misbehave (skewed-future entries
suppress everyone else's toasts). Use `serverTimestamp()` from the RTDB SDK for the
log `timestamp` (and consider it for `attempt.startedAt` — but note countdown display
depends on comparing to local `Date.now()`, so if changed, compute offset once from
`.info/serverTimeOffset`). Acceptance: two clients with deliberately skewed clocks
produce a correctly ordered history and everyone's toasts still fire.

### P2-6: Hatch fill uses Leaflet's private `_path`
`updateAreaLayers` and the no-man's-land polygon set `fill` via `layer._path` — private
API, could break on a Leaflet upgrade and silently leaves claimed zones solid if
`_path` is undefined (e.g. canvas renderer). Guard it: feature-detect `_path`, fall
back to a low-opacity solid fill, and pin the Leaflet version in a comment next to the
CDN URL. Acceptance: with `L_PREFER_CANVAS` forced, zones still visually distinguish
claimed vs locked (fallback), no exceptions.

### P2-7: Normalise action results and error surfacing
`actions.js` functions mostly return `{ok, reason}` but some paths return `{ok:false}`
with no reason, and callers sometimes `showError(res.reason || '')` (empty red box).
Standardise: every failure has a human-readable `reason`; add a tiny shared
`showActionError` in the popup. Acceptance: no code path can show an empty error box.

## P3 — Nice-to-haves (in rough value order)

1. **Attempts-in-progress panel on the Leaderboard** — list zones with `attemptingBy`
   + attempt age, and each team's lockouts (`failedBy`), so the state of the board is
   readable without tapping every zone.
2. **Sound/vibration on big events** — `navigator.vibrate` + a short synth beep
   (WebAudio, no assets) on toast-worthy log entries; respect a mute toggle in
   Settings. Fits the "purposefully hectic" brief.
3. **Site illustration overlay** — `SITE_IMAGE` hook already exists in `map.js`; needs
   the artwork, bounds calibration against `siteBoundary`, and an opacity slider in
   Settings (admin). The illustration is stylised, so expect imperfect registration.
4. **Deploy pipeline** — enable GitHub Pages (Settings → Pages → main/root); document
   the player onboarding URL format `…/CampConquest/?code=<code>`. Optionally a
   `deploy-check` GitHub Action that just validates `areas.js` parses and
   `challenges.csv` has a row for every area (cheap CI guard).
5. **Admin broadcast** — a Settings box letting the admin push a `big` log entry
   (toast to all devices): "lunch at the Village Square in 10".
6. **Countdown pill tap → Leaderboard** for one-tap standings.
7. **Editor QoL** — vertex delete (tap a corner handle twice / long-press), and show
   the site boundary + other zones dimmed while editing for context.

## Known and accepted (do NOT "fix")

- Admin password (`bankofcum`) and reset flow are client-side gates, readable in the
  public repo. Accepted: real security is the secret game code in the DB path + rules;
  players are friends. Do not move auth server-side.
- Results are free text and steal verdicts are self-adjudicated — honour system by
  design.
- Scores/results are hidden by UI gating only; a player reading the Firebase payload
  could cheat. Accepted for this audience.
- The whole game state is a single RTDB node with whole-state transactions — fine at
  this scale (3 writers), do not shard.
- `challenges.csv` parsing is naive TSV (no quoting). Fine; challenge text just must
  not contain tab characters.

## Verification checklist for any change

1. `python -m http.server` + open `http://localhost:8123/?code=<test-code>`.
2. Boot: map renders 21 zones, key panel, no console errors.
3. Two-tab flow: join two teams (name prompts) → start attempt (badge appears in the
   other tab) → third tab blocked from starting → claim with result → steal blind →
   verdict both ways (transfer+lock / lock for owner) → leaderboard scores + bonuses.
4. Admin: unlock, edit an area (owner/result/pass mark), timer start/adjust, reset
   (team+player names must survive after P0-1).
5. Screenshot the map (headless: `msedge --headless=new --screenshot=… <url>`) and
   compare zone layout against the previous commit — geometry must not change unless
   the task is geometry.
